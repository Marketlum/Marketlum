# 012 — Plugin System — Specification

> **Status:** Ready for implementation. Decision trail: [`012-plugin-system-brainstorming.md`](./012-plugin-system-brainstorming.md) (4 rounds, all accepted; deviations noted there).
>
> **Goal:** A first-class plugin system that lets Marketlum be extended — new entities, migrations, API modules, event handlers, scheduled jobs, web pages/nav, and settings — **without editing `@marketlum/core` or `@marketlum/ui` source**. Validated by two first-party plugins: **NBP** (exchange-rate ingestion; backend + settings, no new tables) and **RDHY / RenDanHeYi** (a full-stack new domain slice).

---

## 1. Overview

Today every extension point in Marketlum is a static, hand-maintained list inside the three shipped packages (`@marketlum/core`, `@marketlum/ui`, `@marketlum/shared`); downstream apps generated by `create-marketlum-app` can only consume them. This spec converts those static lists into **open registries fed by an explicit, type-checked `plugins` array** that the downstream app owns.

A plugin is **one npm package** (`@marketlum/plugin-<id>`) with **subpath exports per runtime**:

| Entry | Imports | Provides |
|---|---|---|
| `.` (backend) | `@marketlum/core`, `@nestjs/*`, `typeorm` | `MarketlumApiPlugin` — Nest module, entities, migrations, event providers, scheduled jobs, seeder |
| `/web` (frontend) | `@marketlum/ui`, `react`, `next` | `MarketlumWebPlugin` — nav items, page routes, optional settings component, i18n messages |
| `/shared` (optional) | `zod` | shared identity + settings Zod schema + DTO types |

```
   DOWNSTREAM APP (create-marketlum-app output — thin shell)
   ┌──────────────────────────────────────────────────────────────┐
   │  apps/api  AppModule                                          │
   │    MarketlumCoreModule.forRoot({ plugins: [nbpPlugin] })      │ ← backend seam
   │                                                               │
   │  apps/web  src/plugins.ts                                     │
   │    export const webPlugins = [nbpWebPlugin]                   │ ← frontend seam
   │    app/admin/x/[...slug]/page.tsx  (catch-all → registry)     │
   └──────────────────────────────────────────────────────────────┘
        │ forRoot aggregates                  │ webPlugins feeds
   ┌────▼─────────────────────────┐     ┌─────▼──────────────────────┐
   │ @marketlum/core              │     │ @marketlum/ui              │
   │  • modules → imports[]       │     │  • nav registry (navGroups)│
   │  • entities/migrations →     │     │  • catch-all page resolver │
   │    buildDataSourceOptions()  │     │  • auto settings form      │
   │  • event providers → bus     │     │  • PluginSettingsService   │
   │  • ScheduleModule.forRoot()  │     │    (read) on backend       │
   │  • seeders ← seed:sample     │     │                            │
   └──────────────────────────────┘     └────────────────────────────┘
```

**Locked decisions (Round 1–4):** Option B (typed contract + `forRoot` DynamicModule, not auto-discovery); full-stack capability surface; static `plugins`/`webPlugins` array activation (no runtime DB toggle); first-party in-repo `packages/plugin-*` now, third-party-ready contract later; NBP then RDHY; strict BDD.

## 2. The plugin contract

The contract is **split by runtime** so that `@marketlum/shared` keeps its `zod`/`xstate`-only dependency footprint (no Nest/TypeORM leakage).

### 2.1 Shared identity — `@marketlum/shared` (new `src/plugins/`)

```ts
// packages/shared/src/plugins/plugin-manifest.ts
import { z } from 'zod';

/** kebab-case, ≤ 32 chars, used as the namespace for tables/settings/events. */
export const pluginIdSchema = z
  .string()
  .regex(/^[a-z][a-z0-9-]{1,31}$/, 'plugin id must be kebab-case');

export interface PluginManifest {
  id: string;                    // e.g. "nbp", "rdhy"
  name: string;                  // human label, e.g. "Narodowy Bank Polski"
  version: string;               // the plugin package version (semver)
  marketlumCoreVersion: string;  // semver RANGE this plugin supports, e.g. "^0.5.0"
}

/** A plugin that exposes admin settings declares its schema here (validated on read/write). */
export interface PluginSettingsContract<T = unknown> {
  schema: z.ZodType<T>;
  defaults: T;
}
```

Shared also exports **namespace helpers** used by both runtimes and by the registration guards:

```ts
// packages/shared/src/plugins/namespace.ts
export const pluginTablePrefix   = (id: string) => `plugin_${id.replace(/-/g, '_')}_`;
export const pluginSettingsKey   = (id: string, key: string) => `plugin.${id}.${key}`;
export const pluginEventName     = (id: string, entitySnake: string, verb: string) =>
  `marketlum.plugin.${id}.${entitySnake}.${verb}`;
export const PLUGIN_EVENT_GLOB   = 'marketlum.plugin.**';
```

### 2.2 Backend contract — `@marketlum/core` (new `src/plugins/`)

```ts
// packages/core/src/plugins/marketlum-api-plugin.ts
import type { Type } from '@nestjs/common';
import type { DataSource, EntitySchema, MixedList } from 'typeorm';
import type { PluginManifest, PluginSettingsContract } from '@marketlum/shared';

export interface MarketlumApiPlugin {
  manifest: PluginManifest;

  /** Nest feature module, added to MarketlumCoreModule.imports. */
  module: Type<unknown>;

  /** TypeORM entities owned by the plugin. Table names MUST start with plugin_<id>_. */
  entities?: MixedList<Function | EntitySchema>;

  /** Hand-written migrations owned by the plugin, run after core's in timestamp order. */
  migrations?: MixedList<Function>;

  /** Entity class names (PascalCase) that should emit marketlum.plugin.<id>.* domain events. */
  primaryEntities?: string[];

  /** Settings contract; surfaced in the admin UI and read via PluginSettingsService. */
  settings?: PluginSettingsContract;

  /** Optional sample-data hook, invoked by `pnpm seed:sample` when the plugin is active. */
  seed?: (dataSource: DataSource) => Promise<void>;
}
```

Notes:
- Event **subscribers** are ordinary Nest providers declared inside the plugin's `module` using `@OnEvent('marketlum.**')` — no contract field needed (mirrors `LoggingEventHandler`).
- Scheduled **jobs** are ordinary providers using `@Cron`/`SchedulerRegistry`; core registers `ScheduleModule.forRoot()` once (see §4.3) so plugins need no scheduler wiring of their own.

### 2.3 Frontend contract — `@marketlum/ui` (new `src/plugins/`)

```ts
// packages/ui/src/plugins/marketlum-web-plugin.ts
import type { ComponentType } from 'react';
import type { LucideIcon } from 'lucide-react';
import type { PluginManifest } from '@marketlum/shared';

export interface PluginNavItem {
  slug: string;          // resolves under /admin/x/<slug>
  labelKey: string;      // next-intl key, namespaced plugin.<id>.*
  icon?: LucideIcon;
  group?: string;        // existing core group key, or a new group key
  groupLabelKey?: string;// required only when introducing a NEW group
}

export interface PluginRoute {
  slug: string;                       // matches a PluginNavItem.slug (or a sub-path)
  Component: ComponentType<{ params?: Record<string, string> }>;
}

export interface MarketlumWebPlugin {
  manifest: Pick<PluginManifest, 'id' | 'name'>;
  nav?: PluginNavItem[];
  routes?: PluginRoute[];
  /** Optional custom settings page; when absent the auto-form is rendered from the Zod schema. */
  SettingsComponent?: ComponentType;
  /** next-intl messages, merged under plugin.<id>.* */
  messages?: Record<string /* locale */, Record<string, unknown>>;
}
```

## 3. Backend registration & aggregation

### 3.1 `MarketlumCoreModule.forRoot({ plugins })` → `DynamicModule`

`MarketlumCoreModule` (`packages/core/src/marketlum-core.module.ts`) changes from a static `@Module` to a class exposing a static `forRoot`. The current static decorator content becomes the base; `forRoot` appends plugin contributions.

```ts
export interface MarketlumCoreOptions {
  plugins?: MarketlumApiPlugin[];
}

@Module({})
export class MarketlumCoreModule {
  static forRoot(options: MarketlumCoreOptions = {}): DynamicModule {
    const plugins = options.plugins ?? [];
    validatePlugins(plugins);          // §3.4: id uniqueness, namespace, compat

    return {
      module: MarketlumCoreModule,
      imports: [
        TypeOrmModule.forRoot(buildDataSourceOptions(plugins)),  // §3.2
        EventsModule,
        ScheduleModule.forRoot(),                                // §4.3
        ThrottlerModule.forRoot([{ name: 'default', ttl: 60000, limit: 100 }]),
        ...CORE_FEATURE_MODULES,                                 // today's static list
        PluginsModule.forRoot(plugins),                          // §3.3 + §4.2
        ...plugins.map((p) => p.module),
      ],
      exports: [...CORE_EXPORTED_MODULES, EventsModule, PluginsModule],
      providers: [
        { provide: APP_GUARD, useClass: CsrfProtectionGuard },
        { provide: APP_GUARD, useClass: ThrottlerGuard },
      ],
    };
  }
}
```

`AppModule` (`apps/api/src/app.module.ts`) and the template's `app.module.ts` call `MarketlumCoreModule.forRoot({ plugins })`, importing `plugins` from a new `apps/api/src/plugins.ts`.

### 3.2 `buildDataSourceOptions(plugins)` — single aggregation path

New export in `@marketlum/core` consumed by **both** DataSource definitions, eliminating the duplicated entity/migration arrays.

```ts
// packages/core/src/config/build-data-source-options.ts
export function buildDataSourceOptions(
  plugins: MarketlumApiPlugin[] = [],
): DataSourceOptions {
  return {
    type: 'postgres',
    host: process.env.DATABASE_HOST || 'localhost',
    port: parseInt(process.env.DATABASE_PORT || '5432', 10),
    username: process.env.DATABASE_USERNAME || 'marketlum',
    password: process.env.DATABASE_PASSWORD || 'marketlum',
    database: process.env.DATABASE_NAME || 'marketlum',
    entities:    [...ALL_ENTITIES,    ...plugins.flatMap((p) => asArray(p.entities))],
    migrations:  [...ALL_MIGRATIONS,  ...plugins.flatMap((p) => asArray(p.migrations))],
    synchronize: false,
  };
}
```

- `databaseConfig()` (`packages/core/src/config/database.config.ts`) becomes `(plugins) => buildDataSourceOptions(plugins)`.
- `apps/api/src/data-source.ts` (migration CLI) imports `buildDataSourceOptions` + the app's `plugins` array, so `pnpm migration:run` sees plugin migrations. **Template-sync:** mirror this change into `packages/create-marketlum-app/template/api/src/data-source.ts`.

### 3.3 `PluginsModule`

A small global Nest module that:
- Provides a `PLUGINS` injection token (`MarketlumApiPlugin[]`) so any service can introspect active plugins.
- Provides `PluginSettingsService` (§4.1) and `PluginRegistryService` (validates + exposes manifests, used by the §6 `/plugins` read endpoint and the settings controller).
- Registers the plugin event-subscriber extension (adds each plugin's `primaryEntities` to the emit allow-list — see §5).

### 3.4 Registration guards — `validatePlugins(plugins)`

Run at `forRoot`, throws (fail-fast at boot) on:
1. **Duplicate / reserved id** — ids must be unique and must not collide with a core module/entity name.
2. **Namespace violation** — every plugin entity's `@Entity()` table name must start with `pluginTablePrefix(id)`; every settings key must start with `plugin.<id>.`.
3. **Core-version mismatch** — `semver.satisfies(coreVersion, manifest.marketlumCoreVersion)` is `false`. Error message names the plugin, its required range, and the actual core version.

## 4. Settings

### 4.1 `PluginSettingsService` (backend read/write)

Wraps the existing `SystemSettingsService` (`packages/core/src/system-settings/`), scoping every key to `plugin.<id>.*` and validating against the plugin's Zod schema.

```ts
@Injectable()
export class PluginSettingsService {
  constructor(private readonly settings: SystemSettingsService,
              @Inject(PLUGINS) private readonly plugins: MarketlumApiPlugin[]) {}

  async get<T>(pluginId: string): Promise<T> {
    const plugin = this.requirePlugin(pluginId);
    const raw = await this.settings.get(pluginSettingsKey(pluginId, 'config'));
    const parsed = raw ? JSON.parse(raw) : plugin.settings!.defaults;
    return plugin.settings!.schema.parse(parsed) as T;   // defaults + validation
  }

  async set<T>(pluginId: string, value: T): Promise<void> {
    const plugin = this.requirePlugin(pluginId);
    const validated = plugin.settings!.schema.parse(value);
    await this.settings.set(pluginSettingsKey(pluginId, 'config'), JSON.stringify(validated));
  }
}
```

The whole settings object is stored as a single JSON blob under `plugin.<id>.config` (the `SystemSetting` value column is `text`; key is `varchar(64)` — `plugin.nbp.config` fits comfortably).

### 4.2 Settings UI

- A generic admin page **Settings → Plugins** lists active plugins (from the `/plugins` endpoint, §6) with manifest + last-known status.
- Each plugin's settings render via an **auto-generated form** derived from its Zod schema (reusing the project's `@hookform/resolvers` + Zod pattern), unless the web plugin supplies a `SettingsComponent` (override). NBP supplies a custom component (currency multiselect + cron field + "Refresh now" button); RDHY needs none.
- Persisted through a core endpoint `PUT /plugins/:id/settings` (AdminGuard), which calls `PluginSettingsService.set`.

### 4.3 Scheduling

Core adds `@nestjs/schedule` as a dependency and registers `ScheduleModule.forRoot()` once inside `forRoot` (§3.1). Plugins declare jobs as providers with `@Cron(expr)` or register dynamically via `SchedulerRegistry` when the expression comes from settings (NBP's case). **Template-sync:** add `@nestjs/schedule` to the template's `api/package.json`.

---

## 5. Events

- **Emit:** the existing `DomainEventSubscriber` filters on `PRIMARY_ENTITIES` (`packages/core/src/events/primary-entities.ts`). It is extended to also honor plugin `primaryEntities`, emitting `marketlum.plugin.<id>.<entity_snake>.<verb>` (via `pluginEventName`). `PluginsModule` registers these names at boot.
- **Subscribe:** plugins add `@OnEvent('marketlum.**')` (or a narrower glob) providers inside their `module` — identical to `LoggingEventHandler`. No contract surface needed.
- No outbox/persistence is added (out of scope, §12).

## 6. API surface (core additions)

All under the existing `AdminGuard` (controller-level), mirroring `OfferingsController` etc.

| Method & path | Body / query | Response | Purpose |
|---|---|---|---|
| `GET /plugins` | — | `PluginManifest[]` + `{ hasSettings, status? }` | List active plugins (drives Settings UI + web nav status). |
| `GET /plugins/:id/settings` | — | `{ value: <schema> }` | Current settings (defaults applied). |
| `PUT /plugins/:id/settings` | `<plugin schema>` | `{ value }` | Validate + persist settings. |

Plugin-specific endpoints live in the plugin's own controller, namespaced under `/plugins/:id/...` by convention (e.g. NBP's refresh, §7).

## 7. NBP plugin (`@marketlum/plugin-nbp`)

**Purpose:** ingest exchange rates from Narodowy Bank Polski into the existing `ExchangeRate` table. **No new entities or migrations** — exercises api + settings + scheduling only.

### 7.1 Settings schema (`/shared`)

```ts
export const nbpSettingsSchema = z.object({
  enabled: z.boolean(),
  cron: z.string(),                       // weekdays ~12:30, after NBP publishes
  trackedCurrencies: z.array(z.string()), // ISO codes to ingest, e.g. ["USD","EUR"]
});
export type NbpSettings = z.infer<typeof nbpSettingsSchema>;
```

> **Implemented model (deviation from earlier draft):** tracking is by **ISO code**, not by `Value` id. This lets the summary report a *tracked* code that has no matching `Value` as "skipped" (impossible to express when tracking by id, which presupposes the Value exists). The web multiselect (PR with UI) presents currency `Value`s and stores their codes.

### 7.2 Behavior

- **Source:** `GET https://api.nbp.pl/api/exchangerates/tables/A?format=json` (mid rates, ~33 currencies, one call). HTTP via a thin injectable `NbpClient` (uses global `fetch`), substituted by a fake in tests.
- **Mapping:** for each **tracked ISO code**, find the NBP table-A entry with that `code` (absent ⇒ skip), then the currency `Value` whose `code` equals it (absent ⇒ skip + report). Counter-currency is the PLN `Value` (resolved by `code = 'PLN'`; missing ⇒ run aborts with an error in the summary). **Prerequisites (documented):** currency `Value.code` holds the ISO code; a PLN `Value` exists. Skips are reported in the summary and logged.
- **Storage:** upsert one `ExchangeRate` per `(USD→PLN, mid, effectiveDate)`, tagged `source: 'NBP'`. **Core stores pairs canonically** — the `ExchangeRatesService.create` path runs `canonicaliseRate()` (from `@marketlum/shared`), ordering the pair lexicographically by value ID and inverting the rate when needed; lookups are symmetric. NBP must therefore canonicalise via the same helper (reuse `canonicaliseRate`/`invertRate` from shared) rather than assume raw `from=foreign,to=PLN` storage. The row is keyed on the unique `(fromValueId, toValueId, effectiveAt)`. On conflict, **update the rate only if the existing row's `source = 'NBP'`** (never clobber manual rates) — note the core `create` throws `ConflictException` on duplicate, so NBP needs its own source-guarded upsert (write through the `ExchangeRate` repository, which the plugin obtains via `TypeOrmModule.forFeature([ExchangeRate])` importing the core entity). Decimal handling per the project gotcha: `Number(mid).toFixed(...)`-safe. Behaviour is asserted in BDD via lookups + the source tag, not raw row direction.
- **Schedule:** *(deferred — not in this slice.)* Per strict BDD there is no feature exercising a scheduled run yet, and it needs the `@nestjs/schedule` dependency + `ScheduleModule.forRoot()` in core. When added: a `SchedulerRegistry` cron job (expression from settings) runs the **latest-only** ingest. No backfill (Round 3 Q5). The `cron`/`enabled` settings fields already exist for it.
- **Manual trigger:** `POST /plugins/nbp/refresh` (AdminGuard) runs an immediate latest ingest and returns the summary — the **sole** admin-visible feedback surface (Round 3 Q7):

```json
{ "effectiveDate": "2026-06-25", "updated": ["USD"], "skipped": ["XYZ"], "errors": [] }
```

- **Observability:** silent best-effort otherwise — per-currency skips and API/network errors go to the Nest logger; partial success is allowed (one bad currency never aborts the rest).

### 7.3 Web (`/web`)

Custom `SettingsComponent`: enable toggle, cron input, currency multiselect (fetches currency `Value`s), and a **Refresh now** button that calls `POST /plugins/nbp/refresh` and shows the returned summary. One nav item ("NBP Rates") targeting an existing finance/FX group.

## 8. RDHY plugin (`@marketlum/plugin-rdhy`)

**Purpose:** a **representative vertical slice** of RenDanHeYi that proves plugin-owned entities + migrations, plugin→core FKs, web pages/nav, events, and seeding. The full RDHY domain (VAM, leading targets, two-way contracting) is **deferred to spec 013**.

### 8.1 Entities (plugin-owned, prefixed `plugin_rdhy_`)

| Entity | Table | Key columns | Relationships |
|---|---|---|---|
| `MicroEnterprise` | `plugin_rdhy_micro_enterprises` | `id uuid pk`, `code varchar unique`, `name`, `kind enum(MARKET_ME, NODE_ME, PLATFORM_ME)`, `status enum`, timestamps | `agentId uuid → agents(id)` (one-way FK to core `Agent`), nullable `ON DELETE SET NULL` |
| `EmcAgreement` | `plugin_rdhy_emc_agreements` | `id uuid pk`, `code varchar unique`, `title`, `status enum(DRAFT, ACTIVE, CLOSED)`, `effectiveAt`, timestamps | `valueStreamId uuid → value_streams(id)` (one-way FK to core `ValueStream`); `microEnterpriseId uuid → plugin_rdhy_micro_enterprises(id)` |

> Plugin→core FKs are the proposed mappings (Round 4 Q2, not elaborated by the user) — **confirm exact target columns/semantics during PR 2 implementation** before writing the migration.

`@marketlum/shared` (or the plugin's `/shared` entry) holds the `create/update/response` Zod schemas + inferred types for both entities, following the per-domain-file + named-export pattern.

### 8.2 Migrations

One hand-written migration in the plugin (`packages/plugin-rdhy/src/migrations/<ts>-CreateRdhyTables.ts`) creating both tables, the FKs, and the `code` unique indexes. Concatenated after core migrations by `buildDataSourceOptions`. `synchronize: false` honored.

### 8.3 API

Two standard CRUD modules (controller + service + `ZodValidationPipe`, AdminGuard), entities registered via `TypeOrmModule.forFeature`. Both entity names go in the plugin's `primaryEntities`, so CRUD emits `marketlum.plugin.rdhy.micro_enterprise.*` / `...emc_agreement.*`.

### 8.4 Events (both directions)

- **Subscribe:** an `@OnEvent('marketlum.agent.created')` provider (demonstrates reacting to core) — e.g. log/flag a candidate ME for the new agent (no auto-mutation in the pilot).
- **Emit:** via §5 plugin primary-entities.

### 8.5 Web (`/web`)

List + detail + form pages for each entity (reusing `@marketlum/ui` table/form primitives), exposed as `PluginRoute`s at slugs `micro-enterprises` and `emc-agreements`, resolved under `/admin/x/<slug>`. Nav items declare a **new group** "RenDanHeYi". i18n messages under `plugin.rdhy.*`.

### 8.6 Seed

`seed?(dataSource)` inserts ~2 MEs + 1 EMC agreement linked to seeded core agents/value streams, run by `pnpm seed:sample` after core seeding.

## 9. Web app wiring (apps/web) & nav registry

- **`packages/ui/src/layouts/admin-layout.tsx`:** the hardcoded `navGroups[]` becomes `coreNavGroups` merged with plugin nav. Merge rule: a `PluginNavItem` with an existing `group` key appends into that core group; a new `group` key creates a group (ordered after core groups). Labels resolve via next-intl. A new `PluginRegistryProvider` (React context) supplies `webPlugins` to the layout and the catch-all route.
- **`apps/web/src/plugins.ts`** (new): `export const webPlugins: MarketlumWebPlugin[] = [ /* nbpWebPlugin */ ]`. Imported by the providers.
- **`apps/web/src/app/admin/x/[...slug]/page.tsx`** (new): client component that reads `slug`, looks up the matching `PluginRoute` in `webPlugins`, and renders its `Component`; 404s on miss. The reserved `x/` segment cannot collide with core static routes.
- **i18n:** plugin `messages` merged into the next-intl dictionary at the web app's message-loading boundary.

**Template-sync (CLAUDE.md):** every `apps/web` and `apps/api` change above must be mirrored under `packages/create-marketlum-app/template/`:
- `template/api/src/app.module.ts` → `forRoot({ plugins })`, `template/api/src/plugins.ts` → `export const plugins = []`, `template/api/src/data-source.ts` → `buildDataSourceOptions`, `template/api/package.json` → `@nestjs/schedule`.
- `template/web/src/plugins.ts` → `export const webPlugins = []`, `template/web/src/app/admin/x/[...slug]/page.tsx` → catch-all, plus `PluginRegistryProvider` wiring.
- Pilot plugins are **not** pre-wired in the template (Round 4 Q6) — generated apps are plugin-ready but lean.

## 14. BDD coverage (strict)

Tests travel with the code (Round 4 Q5). jest-cucumber `loadFeature`/`defineFeature`, ref-counted shared app, `createAuthenticatedUser()`.

**Core mechanics** (`packages/core` / under PR 1):
- `plugins/registration.feature` — `forRoot` aggregates a fake plugin's module/entities; `/plugins` lists it. (~3)
- `plugins/validation.feature` — duplicate id rejected; namespace violation rejected; core-version mismatch rejected. (~3)
- `plugins/settings.feature` — get returns defaults; put validates + persists; invalid payload 422. (~3)
- `plugins/events.feature` — a plugin `primaryEntities` entry emits `marketlum.plugin.<id>.*`. (~2)

**NBP** (`packages/plugin-nbp`, PR 1):

## 15. Out of scope

- **Runtime enable/disable / DB-driven toggles** — activation is the static `plugins`/`webPlugins` arrays (Round 2 Q1, Round brainstorming lock).
- **Auto-discovery of plugins from `node_modules`** — explicitly rejected (Option B).
- **Event outbox / cross-process or durable event delivery** — events stay in-process EventEmitter2 (§5).
- **Full RenDanHeYi domain** (VAM, leading targets, two-way contracting, platforms) — deferred to **spec 013** (Round 4 Q1).
- **NBP historical backfill, bid/ask (Table C), persisted sync history/events** — (Round 3 Q5/Q2/Q7).
- **Plugins extending/altering core entities or core referencing plugin entities** — one-way dependency only (Round 4 Q2).
- **Third-party plugin marketplace / signing / sandboxing** — the contract is built third-party-*ready*, but distribution hardening is a later phase.
