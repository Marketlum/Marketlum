# 012 вҖ” Plugin System вҖ” Brainstorming

> **Goal:** Add a first-class plugin system so Marketlum can be extended (new entities, migrations, API modules, event handlers, web pages/nav, settings) **without editing `@marketlum/core` / `@marketlum/ui` source**. Validate the design with two first-party plugins: **NBP** (loads exchange rates from Narodowy Bank Polski; backend + settings, no new tables) and **RenDanHeYi / RDHY** (adds micro-enterprises, EMC agreements, etc.; full-stack new domain).

## Context

**What exists today (mapped from the codebase):**

Everything ships through three publishable packages; `apps/*` are thin shells.

| Package | Role |
|---|---|
| `@marketlum/core` | entire NestJS backend вҖ” ~28 feature modules, all entities, all migrations, events, seeders |
| `@marketlum/ui` | entire Next.js frontend вҖ” admin pages, layouts, UI primitives, i18n |
| `@marketlum/shared` | framework-agnostic вҖ” Zod schemas, enums, xstate machines, domain-event types (deps: `zod`, `xstate` only) |

**Every extension point is a static, hand-maintained list inside the shipped packages.** A downstream app generated by `create-marketlum-app` can only *consume* these; it cannot *contribute* to them.

| Concern | Current mechanism | File |
|---|---|---|
| Backend modules | static `imports[]` / `exports[]` | `packages/core/src/marketlum-core.module.ts` |
| Entities | static `ALL_ENTITIES[]` | `packages/core/src/entities.ts` |
| Migrations | static `ALL_MIGRATIONS[]` | `packages/core/src/migrations/index.ts` |
| DataSource (run + CLI) | both import the two arrays above | `config/database.config.ts` **and** `apps/api/src/data-source.ts` (duplicated) |
| Events (emit) | TypeORM subscriber + `PRIMARY_ENTITIES[]` | `events/domain-event.subscriber.ts` |
| Events (subscribe) | `@OnEvent('marketlum.**')` provider | `events/logging-event.handler.ts` (pattern) |
| Web nav | hardcoded `navGroups[]` literal in a component | `packages/ui/src/layouts/admin-layout.tsx` |
| Web routes | thin re-export `page.tsx` files | `apps/web/src/app/admin/**` |
| Settings | DB key/value (`varchar(64)` вҶ’ `text`) | `system-settings/` |

**Useful grounding for the two pilot plugins:**
- `ExchangeRate` entity already has a `source: text` column and a unique `(fromValueId, toValueId, effectiveAt)` constraint вҖ” NBP can upsert rows tagged `source = 'NBP'`. Currencies are `Value` rows referenced by UUID.
- `SystemSetting` is a flat `key (varchar 64) вҶ’ value (text)` store вҖ” the natural seam for plugin config.
- `MarketlumCoreModule` is currently a plain static `@Module` (no `forRoot`). `AppModule` just imports it + `ConfigModule`.

## Round 1 вҖ” Foundations: contract shape & registration mechanics

This round nails down how a plugin is physically packaged and how its backend contributions (modules, entities, migrations) get aggregated into both the running app and the migration CLI. The frontend nav/route mechanics and the per-plugin shapes (NBP scheduling, RDHY domain) come in later rounds.

**Q1. How is a single plugin physically packaged, given the core/ui/shared runtime split?**
A plugin has backend code (Nest/TypeORM вҖ” must NOT be bundled into Next) and frontend code (React/Next).
- [x] **One npm package, subpath exports per runtime** вҖ” `@marketlum/plugin-nbp` (backend entry `.`) + `@marketlum/plugin-nbp/web` (frontend entry). One version, one publish; mirrors the existing core-vs-ui split. The app imports backend from `.`, the web app imports pages from `/web`.
- [ ] **Three packages per plugin** (`-core`, `-ui`, `-shared`) вҖ” cleanest isolation, but 3Г— publishing/versioning overhead per plugin.
- [ ] **Single mixed entry** вҖ” simplest to author, but forces Next to bundle Nest/TypeORM backend code; fragile.

**Answer:**

**Q2. Where do the contract TYPES live?**
The contract spans runtimes, and `@marketlum/shared` currently only depends on `zod`/`xstate` (no Nest/TypeORM).
- [x] **Split by runtime** вҖ” `MarketlumApiPlugin` (module + entities + migrations + event providers) typed in `@marketlum/core`; `MarketlumWebPlugin` (nav items + page routes) typed in `@marketlum/ui`; shared identity (`id`, `version`, settings Zod schema) in `@marketlum/shared`. Each side imports only what it needs; keeps Nest/TypeORM out of shared.
- [ ] **One monolithic `MarketlumPlugin` in `@marketlum/shared`** вҖ” single import, but drags Nest + TypeORM types into the framework-agnostic package.
- [ ] **Everything in `@marketlum/core`** вҖ” ui would import types from core (it currently imports only from shared); adds a new coupling direction.

**Answer:**

**Q3. Backend registration entry point.**
- [x] **`MarketlumCoreModule.forRoot({ plugins })` вҶ’ `DynamicModule`** вҖ” aggregates plugin modules into `imports`, entities/migrations into the DataSource, event providers into the bus. Single registration call in `AppModule`. Converts today's static `@Module` to a dynamic one.
- [ ] **Separate `PluginsModule.forRoot(plugins)`** imported alongside a still-static `MarketlumCoreModule` вҖ” smaller change to core, but two registration points and entities still need a second path into the DataSource.

**Answer:**

**Q4. How do plugin entities + migrations reach BOTH the running app and the migration CLI?**
Today `databaseConfig()` and `apps/api/src/data-source.ts` independently import the two static arrays.
- [x] **Shared `buildDataSourceOptions(plugins)` helper** вҖ” exported from core, used by both `databaseConfig()` and `data-source.ts`. Final entity/migration lists = core defaults + `plugins.flatMap(p => p.entities/p.migrations)`. The app passes its `plugins` array to both. One aggregation path, no hand-edited arrays.
- [ ] **Keep `ALL_ENTITIES`/`ALL_MIGRATIONS` as the only path; plugins documented to append manually** вҖ” defeats the purpose; rejected.
- [ ] **Glob/auto-discovery of plugin entity files** вҖ” fights TypeORM's static, bundler-unfriendly loading; rejected.

**Answer:**

**Q5. Migration ownership for plugin-owned tables (RDHY).**
The project's gotcha: "write manual migrations for clean results", `synchronize: false` everywhere.
- [x] **Each plugin ships its own hand-written migrations** inside its package, surfaced via `plugin.migrations`, concatenated after core migrations in timestamp order. Plugin migrations live and version with the plugin.
- [ ] **Plugin tables created via `synchronize`/auto-generate** вҖ” contradicts the project's migration discipline; rejected.
- [ ] **Core owns all migrations, including plugin tables** вҖ” couples core to every plugin's schema; rejected.

**Answer:**

**Q6. Namespacing, to keep plugins from colliding with core or each other (matters once third-parties ship).**
- [x] **Convention-enforced prefixes** вҖ” tables `plugin_<id>_*` (e.g. `plugin_rdhy_micro_enterprises`); settings keys `plugin.<id>.<key>`; plugin event names `marketlum.plugin.<id>.<entity>.<verb>`. Validated at registration; reserved-id check prevents shadowing core entities.
- [ ] **No enforced namespacing**, rely on plugin authors вҖ” lighter, but collision risk and no guard rails for the third-party phase.

**Answer:**

**Q7. Plugin identity & core-version compatibility.**
- [x] **Manifest `{ id, name, version, marketlumCoreVersion: <semver range> }`; core checks the range at `forRoot` and throws on mismatch** вҖ” cheap guard that prevents silent breakage when a downstream app upgrades core but not a plugin.
- [ ] **Minimal `{ id, version }`, no compat check** вҖ” less ceremony, but upgrades can break plugins silently.

**Answer:**

---

## Round 1 вҖ” Accepted

All seven recommendations accepted unchanged: (1) one package + subpath exports, (2) contract split by runtime across core/ui/shared, (3) `MarketlumCoreModule.forRoot({ plugins })` DynamicModule, (4) shared `buildDataSourceOptions(plugins)` helper for both DataSources, (5) plugins ship their own migrations, (6) enforced `plugin_<id>_*` / `plugin.<id>.*` namespacing, (7) manifest with `marketlumCoreVersion` compat check.

**Q1. How does the web app learn which plugins are active (frontend side)?**
The backend has its `plugins` array; the frontend is a separate build and can't import Nest/TypeORM code.
- [x] **Explicit `webPlugins` array in the app** вҖ” a single `apps/web/src/plugins.ts` (mirrored in the template) importing each plugin's `/web` entry, parallel to the backend `plugins` array. Two short lists, both static and type-checked; no runtime coupling.
- [ ] **Fetch an `/api/plugins` manifest at runtime** вҖ” one source of truth, but Next can't generate routes from runtime data, so routing still needs a static path; adds a load-time dependency.
- [ ] **Single combined array re-exported to both runtimes** вҖ” one edit site, but forces the frontend bundle to touch backend plugin code; fragile.

**Answer:**

**Q2. How do plugin pages enter the App Router, given build-time routing + the thin-shell template?**
- [x] **One catch-all route under a reserved segment** вҖ” `apps/web/src/app/admin/x/[...slug]/page.tsx` looks the page component up in `webPlugins` by slug. Plugins add pages with **zero** new route files in the app/template; URLs look like `/admin/x/micro-enterprises`. Safe: the reserved `x/` prefix can never collide with core's static routes.
- [ ] **Root-level optional catch-all** `app/admin/[[...slug]]/page.tsx` вҖ” cleaner URLs (`/admin/micro-enterprises`), and Next gives static core routes precedence, but the catch-all owns all 404s and unknown-path handling under `/admin`; riskier.
- [ ] **Per-page thin re-export files** (today's pattern) вҖ” explicit and debuggable, but every plugin page is a new file in the app **and** the create-marketlum-app template; manual sync per page.

**Answer:**

**Q3. Where do plugin nav entries land in the sidebar?**
`navGroups[]` becomes data-driven (core groups + plugin contributions).
- [x] **Plugin nav items target a group by key, or define their own group** вҖ” a plugin may append into an existing core group (e.g. `finance`) or declare a new group; registry merges, plugin groups ordered after core. Maximum flexibility (NBP slots a "FX Rates / NBP" item into an existing group; RDHY gets its own group).
- [ ] **All plugin nav under one fixed "Extensions" group** вҖ” simplest merge logic, but everything plugin-related is quarantined in one section regardless of domain fit.

**Answer:**

**Q4. How are plugin settings surfaced in the admin UI?**
Each plugin ships a settings Zod schema (Round 1 Q2).
- [x] **Auto-generated form, with optional custom-component override** вҖ” core/ui renders the plugin's Zod schema as a form on a generic per-plugin settings page by default; a plugin may supply its own React component when the auto-form isn't enough. Free settings UI for NBP, full control for complex cases.
- [ ] **Auto-generated form only** вҖ” least code, but can't express anything the schema-to-form can't (e.g. NBP's "pick which currency pairs to track" multiselect against live `Value` data).
- [ ] **Each plugin ships its own settings page** вҖ” total control, but every plugin reimplements form/persist boilerplate.

**Answer:**

**Q5. How does a plugin READ its settings on the backend at runtime?**
Persisted in `system-settings` under `plugin.<id>.*` (Round 1 Q6).
- [x] **Typed `PluginSettingsService` injected into plugin modules** вҖ” wraps `system-settings`, scoped to the plugin's namespace, validates against the plugin's Zod schema on read/write. NBP reads enabled pairs + schedule through it; one tested accessor.
- [ ] **Plugins read raw `SystemSettingsService` directly** вҖ” no new abstraction, but each plugin re-implements key-prefixing + validation; easy to drift.

**Answer:**

**Q6. Plugin UI i18n / labels.**
Core ui uses `next-intl` with namespaced messages.
- [x] **Plugins ship their own message catalogs under `plugin.<id>.*`** вҖ” the web app merges them into the next-intl dictionary; plugin nav labels and pages are translatable without editing core message files.
- [ ] **Plain inline English strings for now** вҖ” fastest to build the pilots; defer i18n until the third-party phase.

**Answer:**

## Round 3 вҖ” NBP plugin: scheduling, fetch & rate ingestion

The NBP plugin is the **backend-only pilot** that proves the api + settings + scheduling capabilities without any new tables. It pulls rates from Narodowy Bank Polski and upserts `ExchangeRate` rows tagged `source = 'NBP'`. Grounding: `ExchangeRate` is `(fromValueId, toValueId, rate, effectiveAt, source)` with a unique `(fromValueId, toValueId, effectiveAt)` constraint; currencies are `Value` rows with a unique `code`. NBP table A is **PLN-quoted** (`1 unit of foreign = N PLN`).

**Q1. Scheduling вҖ” how does NBP run automatically? (no `@nestjs/schedule` in the repo today)**
- [x] **Core adds `@nestjs/schedule` and registers `ScheduleModule.forRoot()` once; plugins use `@Cron`/`SchedulerRegistry` in their own providers** вҖ” scheduling becomes a reusable plugin capability; NBP registers a daily job whose cron expression comes from plugin settings. One scheduler, no conflicts.
- [ ] **Each plugin bundles its own `ScheduleModule`** вҖ” self-contained, but multiple `forRoot()` registrations risk registry conflicts.
- [ ] **No scheduler; refresh only via manual trigger or an external cron hitting an endpoint** вҖ” simplest, but not "automatic"; pushes scheduling onto ops.

**Answer:**

**Q2. Which NBP API and rate type?**
NBP REST API (`https://api.nbp.pl/api/...`, `?format=json`). Table A = average/mid rates (~33 currencies); B = less-common; C = bid/ask.
- [x] **Table A, mid rate, whole-table fetch** вҖ” `GET /exchangerates/tables/A?format=json` returns all currencies in one call; store the single mid `rate`. Matches `ExchangeRate`'s single `rate` column; one request per day.
- [ ] **Tables A + C (adds bid/ask)** вҖ” richer, but `ExchangeRate` has only one `rate` column, so bid/ask can't be stored without a schema change (and NBP plugin owns no tables in this pilot).
- [ ] **Per-currency endpoint `/rates/A/{code}`** вҖ” one call per tracked currency; only worth it when tracking a handful.

**Answer:**

**Q3. Currency вҶ’ `Value` mapping, and which pairs get rates.**
NBP returns ISO codes (`USD`, `EUR`); the PLN counter-currency must exist as a `Value`.
- [x] **Match NBP ISO code to `Value.code`; only existing, settings-enabled currencies are ingested; counter-currency is the PLN `Value`** вҖ” admin picks which currencies to track (multiselect over currency `Value`s); unmapped/disabled codes are skipped and logged. Requires currency `Value.code` to hold the ISO code (documented prerequisite; PLN `Value` required).
- [ ] **Explicit per-code mapping table in settings** (NBP `USD` вҶ’ chosen `Value` id) вҖ” works even when `Value.code` isn't the ISO code, but more setup per currency.
- [ ] **Auto-create missing currency `Value`s for every NBP code** вҖ” zero setup, but lets an external feed mint core domain rows; too invasive.

**Answer:**

**Q4. Storage direction, idempotency, and coexistence with manual rates.**
- [x] **Upsert `from=<foreign> to=PLN rate=<PLN-per-unit> effectiveAt=<NBP effectiveDate> source='NBP'`, keyed on the unique `(from,to,effectiveAt)`; on conflict, update the rate only if the existing row's `source='NBP'`** вҖ” re-runs are idempotent and NBP never clobbers manually-entered rates. NBP's native quote direction is preserved.
- [ ] **Always overwrite on conflict regardless of source** вҖ” simpler, but a daily job could silently stomp a hand-corrected rate.
- [ ] **Insert-only, skip on any conflict** вҖ” never overwrites, but can't correct a previously-published NBP value that NBP later revised.

**Answer:**

**Q5. Historical backfill vs latest-only.**
NBP supports date-range queries (`/tables/A/{start}/{end}`, max ~367 days).
- [ ] **Latest-only by default, with an on-demand date-range backfill** вҖ” the daily cron ingests the newest table; an admin action (with a start/end) backfills history when needed. Keeps steady-state cheap, history available when wanted.
- [ ] **Always backfill a rolling window (e.g. last 30 days) each run** вҖ” self-healing against missed days, but redundant fetches every run.
- [x] **Latest-only, no backfill at all** вҖ” simplest, but gaps from downtime never fill.

**Answer:**

**Q6. Manual refresh trigger.**
- [x] **An admin-only endpoint + a "Refresh now" button on the NBP settings page** вҖ” triggers an immediate fetch (latest or a chosen range), returning a summary (currencies updated, skipped, errors). Good for first-run and debugging.
- [ ] **No manual trigger; rely on the schedule only** вҖ” less surface area, but no way to force a refresh or run the initial load on demand.

**Answer:**

**Q7. Failure handling & observability when a run fails or a currency can't be mapped.**
- [ ] **Per-run summary persisted + emitted as a domain event; partial success is allowed** вҖ” unmapped/disabled codes are skipped (not fatal), network/API errors are logged and surfaced on the settings page (last-run status + timestamp); the run emits `marketlum.plugin.nbp.sync.completed/failed`. Observable without new tables (status in settings/log).
- [ ] **Fail-fast: any error aborts the whole run** вҖ” strict, but one missing currency would block all others.
- [x] **Silent best-effort, log only** вҖ” least code, but failures are invisible to the admin.

**Answer:**

## Round 3 вҖ” Accepted (with two changes)

Accepted as recommended: Q1 scheduler added to core (`@nestjs/schedule`), Q2 Table A whole-table mid-rate fetch, Q3 ISO-codeвҶ’`Value.code` mapping with enabled-currency multiselect (PLN `Value` + ISO codes are prerequisites), Q4 source-guarded idempotent upsert, Q6 manual "Refresh now" endpoint+button. **Changed by user:** Q5 вҶ’ **latest-only, no backfill**; Q7 вҶ’ **silent best-effort, log only** (no persisted summary / sync events). Consequence carried into the spec: the manual-refresh endpoint's response (updated/skipped/errors) is the sole admin-visible feedback surface.

## Round 4 вҖ” RDHY full-stack pilot, delivery, template sync & BDD

NBP proved api+settings+scheduling with no tables. **RDHY proves the rest of the contract**: plugin-owned entities + migrations, pluginвҶ’core foreign keys, web pages/nav, events, and seeding. This round also scopes RDHY's domain depth (this is the *plugin-system* spec, not the RDHY domain spec) and pins down delivery, template sync, and test coverage. (RenDanHeYi terms: **ME** = micro-enterprise, an autonomous unit; **EMC** = Ecosystem Micro-Community; agreements bind them.)

**Q1. How much RDHY domain do we design in THIS spec?**

**Q2. How do RDHY (plugin) entities relate to existing core entities?**
This validates the most important structural capability: can a plugin reference core?
- [x] **Plugin entities may hold FKs to core entities; core never references plugin entities (one-way)** вҖ” e.g. `MicroEnterprise.agentId вҶ’ Agent`, `EmcAgreement.valueStreamId вҶ’ ValueStream`, by importing the core entity class. Proves pluginвҶ’core relationships without touching core schema. (Exact mappings to confirm in the Answer / spec.)
- [ ] **RDHY entities fully standalone, no core FKs** вҖ” simpler, but doesn't prove the (essential) pluginвҶ’core relationship.
- [ ] **Extend/alter core entities (add columns to `Agent` etc.)** вҖ” powerful, but violates "no editing core"; rejected.

**Q3. Does RDHY exercise the events capability, and in which direction?**
- [x] **Both directions** вҖ” RDHY subscribes to a core event (e.g. `marketlum.agent.created`) via an `@OnEvent` provider, and emits its own `marketlum.plugin.rdhy.*` events for its entities (added to the plugin's `PRIMARY_ENTITIES`-equivalent). Proves subscribe + emit.
- [ ] **One direction only** (emit *or* subscribe) вҖ” proves half the capability.
- [ ] **No events in the pilot** вҖ” leaves the events capability unvalidated end-to-end.

**Q4. Seed data вҖ” can a plugin contribute sample data?**
`pnpm seed:sample` is core-owned today.
- [x] **Plugin contract gains a `seed?(dataSource)` hook; `seed:sample` runs active plugins' seeders after core's** вҖ” RDHY seeds a couple of MEs + an EMC agreement; proves plugins can seed without editing core's seed command.
- [ ] **Add RDHY seed rows to core's `seed-sample.command.ts`** вҖ” works, but violates "no editing core".
- [ ] **No plugin seed data** вҖ” pilot ships empty; weaker demo and harder manual testing.

**Q5. Where do plugin tests live (strict BDD is mandatory)?**
